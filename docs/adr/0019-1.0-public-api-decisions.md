# ADR-0019: 1.0 public API decisions

- **Status:** Accepted
- **Date:** 2026-07-08
- **Deciders:** Project owner
- **Amends:** [ADR-0005](0005-domain-modelling-and-strong-typing.md) (drops the `serde` feature), [ADR-0009](0009-no_std-and-optional-libm.md) (the feature set), [ADR-0014](0014-transcendental-single-sum-operations.md) (single-sum module path)

## Context

Publishing `time_value` `1.0.0` fixes the public API as a compatibility promise:
after it ships, a breaking change needs `2.0`. A review of the surface before the
first release turned up four things worth settling now, while they are still free
to change. (Additive changes — new methods, new trait impls, new `#[non_exhaustive]`
error variants, new features — stay available in the `1.x` line and are out of
scope here.)

## Decision

**1. Drop the `serde` feature.** [ADR-0005](0005-domain-modelling-and-strong-typing.md)
planned feature-gated `serde` derives, but the `1.0` code never wired them: the
feature pulled in the `serde` dependency and derived nothing, and neither binary
uses it (the CLI/MCP marshal plain `f64` via their own `serde_json`). A dead,
misleading feature is worse than none. Remove it. Re-introducing it later — with a
deliberate `#[serde(transparent)]` representation so the newtypes serialise as
bare numbers rather than structs carrying a `PhantomData` — is a non-breaking
`1.x` addition, to be done when a real consumer needs it.

**2. `Money`'s finiteness guarantee is on *construction*, not on operation
results.** [`Money::new`] rejects `NaN`/`±∞`, so a constructed `Money` is finite.
But the operations build their results with an internal unchecked constructor, so
on extreme inputs their `f64` arithmetic can overflow to `±∞`. Rather than make
every operation fallible (a large ergonomic cost for an overflow that needs
absurd inputs, and one that could not be walked back post-`1.0`), the operations
stay infallible and return `Money`, and the documentation states the narrower,
honest guarantee. Checked `try_*` variants remain a possible non-breaking
addition.

**3. Single-sum operations live in a `single_sum` module, mirroring `annuity`.**
Previously `present_value` / `future_value` were re-exported at the crate root
while the annuity functions sat in a module — an asymmetry, and the two pairs
share names. Promote `single_sum` to a public module and drop the root
re-exports, so the taxonomy is parallel and self-describing:
`single_sum::present_value`, `annuity::present_value`,
`Cashflows::net_present_value`.

**4. `internal_rate_of_return_from` keeps its `f64` seed.** It is the one place a
rate-like value is a plain `f64` rather than a `Rate<P>`, but deliberately: the
argument is an *initial guess* for the solver, not a validated rate. Keeping it
`f64` lets a caller pass any starting estimate without constructing a `Rate`, and
the iteration already guards the valid domain (it fails with `IrrDidNotConverge`
if it leaves it). The method still *returns* a validated `Rate<P>`.

## Consequences

- `--features serde` no longer exists; the default and `--all-features` builds are
  unaffected in behaviour (nothing used it). The core's only optional dependency
  is now `libm`.
- `time_value::present_value` / `future_value` become
  `time_value::single_sum::present_value` / `future_value`; the CLI and MCP call
  sites move to the module path (which also removes a name clash with the MCP
  server's own `present_value` / `future_value` tool methods).
- `Money`'s docs no longer over-promise; callers feeding potential-overflow
  magnitudes are told to check `value().is_finite()`.
- These are the last free changes to these corners of the surface; further changes
  after `1.0.0` publishes require a major version.

## Alternatives considered

- **Wire `serde` now** rather than dropping it — reasonable, but commits to a
  serialisation format at `1.0` with no consumer to validate it against; deferring
  keeps that decision for when it is informed by real use.
- **Make the operations fallible** (return `Result<Money>`) to preserve a total
  finiteness guarantee — correct but heavy; overflow requires unrealistic inputs,
  and `try_*` variants can be added later without a break.
- **Keep single-sum at the crate root** and instead move `annuity` to the root —
  the names collide (`present_value` for both), so both-at-root is impossible;
  both-as-modules is the only symmetric option.
- **Take a `Rate<P>` seed for the IRR solver** — more type-consistent, but conflates
  a validated rate with a mere starting guess and adds ceremony for the power-user
  method that needs it least.
